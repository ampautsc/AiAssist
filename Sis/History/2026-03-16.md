## Session 2: DnD Data File Comprehensive Inventory (Pure Research)

### What Was Done
Conducted a complete inventory of ALL D&D reference/content data files across the entire codebase — dnd-builder (server + client), DnD/data/, DnD/combat-ai/data/, DnD/utils/. This was pure research with zero code changes.

### Key Findings
- **10 categories** of data across 3 major directory trees
- **dnd-builder server/combat/data/**: 35 spells, 16 creature templates, 15 loot tables (JS module.exports)
- **dnd-builder server/data/**: 10 items (JSON), 71 species (JSON), 32 NPC personalities (JSON), 1 town (JSON), 71 species HTML pages
- **dnd-builder seed files**: ~250+ inline data entries seeded to MongoDB (feats, items, builds, spell notes, class features, skills, backgrounds, level progression, conditions)
- **dnd-builder frontend**: 7 encounters, 14 terrain types, HUD mock data (ES modules)
- **DnD/data/**: 12 classes, 9 races, 78 feats, 12 backgrounds, 26 weapons, 4 subclass trees (JSON) + 2 empty stubs
- **DnD/combat-ai/data/creatures/**: 11 YAML creature stat blocks
- **AI tactics**: 10 combat profiles + 4 strategy profiles
- 12 Mongoose model schemas define the data shapes

### What Was Surprising
- Two completely separate creature data systems: JS templates (combat engine) and YAML stat blocks (Python combat-ai) with overlapping but different creatures
- Two separate spell data systems: combat spells.js (35 mechanical) vs seed-reference.js BARD_SPELL_NOTES (~100 tactical analysis)
- DnD/data/spells.json and equipment.json are empty stubs — all real data lives elsewhere
- NPC personality files have a rich "consciousness context" schema (innerMonologue, emotionalBaseline, contradictions) beyond standard D&D stat blocks

### Lessons
- Data is spread across 3 ecosystems: JS (dnd-builder), JSON (DnD/data/), YAML (combat-ai) — any consolidation effort must account for all three
- Seed files contain significant inline reference data that isn't visible from file names alone — always read seed files during inventory
