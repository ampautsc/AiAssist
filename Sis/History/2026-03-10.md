# History — 2026-03-10

## Interaction: Implement D&D 5e Combat AI Tactics Engine

### What Was Discussed
Creator requested implementation of a new `tactics.js` file for the D&D 5e combat simulation system. The file needed to pass all tests in `tactics.test.js` (151 tests covering battlefield assessment, evaluators, profiles, decision engine, reaction evaluators, and monster AI).

### What Was Done

#### tactics.js — Complete Rewrite
Implemented a priority-based tactical AI system with:

1. **`assessBattlefield(creature, allCombatants, round)`** — builds a context snapshot with enemies, allies, activeEnemies, enemiesInMelee, charmedAllies, hpPct, isFlying, isInvisible, canFly

2. **Targeting helpers** — `selectHighestThreat` (prefers casters), `selectWeakest` (lowest HP), `selectClosestCharmedAlly`

3. **Evaluator functions** for each archetype:
   - Lore Bard: survival invisibility, opening AoE, concentration management, self-heal bonus action
   - Cult Fanatic: invisible fallback, flying target, spiritual weapon, shake awake, melee/spell
   - Monster archetypes: dragon breath, giant rock throw/melee, mage fireball/firebolt/misty step, archmage cone of cold, lich power word stun/finger of death/cloudkill
   - Shared: dodge, ranged cantrip with approach

4. **Reaction evaluators**: cutting words (bardic inspiration), counterspell (dangerous spells), legendary resistance

5. **`makeDecision()`** — runs evaluators in priority order, collects `_bonusActionOnly` results and merges their bonus actions into the main decision

6. **Profile registry** — 10 profiles (lore_bard, cult_fanatic, generic_melee/ranged, dragon, giant_bruiser, mage_caster, archmage_caster, lich_caster, undead_melee)

7. **Factory functions** — `makeTacticalAI(profileMap)` and `makeReactionAI(profileMap)` for encounter runners

#### encounterRunner.js — Complete Rewrite
Discovered that the encounterRunner also needed a complete rewrite to support the new API format used by all tests:

- **New API**: `runEncounter({ combatants, getDecision, maxRounds, verbose })` → `{ winner, rounds, log, analytics, positionSnapshots }`
- New exports: `rollInitiative`, `resetTurnState`, `processStartOfTurn`, `processEndOfTurnSaves`, `checkVictory`, `resolveWeaponAttack`, `resolveBreathWeapon`, `resolveShakeAwake`, `resolveDragonFear`, `buildAnalytics`
- Proper concentration save checking on weapon hits (calls `mech.breakConcentration` on failed saves)
- Turn processing: initiative → reset state → start-of-turn effects → get decision → resolve action + bonus → end-of-turn saves → victory check
- Victory conditions: party/enemy dead, stalemate at round 15+ with incapacitated enemies, draw on timeout

### What Was Learned

**Key technical insights:**
- Vicious Mockery is in `cantrips` array, not `spellsKnown`, so `evalFallbackCantrip` must check both
- lich_caster profile needs `evalMageFireball` before `evalMageFireBolt` (not just FireBolt) — test expected Fireball fallback when high slots spent but level 3 still available
- The `_bonusActionOnly` pattern allows evaluators like `evalConcentrationSelfHeal` and `evalMageMistyStep` to augment another evaluator's main decision with a bonus action
- The encounterRunner was already being used with the new API in `scenarioHarness.js` but hadn't been rewritten yet

## Interaction: Architectural Refactor — Separate AI Intent from Engine Target Resolution

### What Was Discussed
Creator identified a fundamental architecture issue: the AI (tactics.js) was deciding WHO gets hit by AoE spells, when that should be the game engine's job. The AI should declare intent (WHAT spell, WHERE to center it), and the engine should determine WHO is affected based on geometry.

Creator's exact insight: "The spell really shouldn't get to decide who is impacted. The game asks what the entity wants to do on their turn, and the entity replies with Cast, Hypnotic Pattern, Centered on Hex68... Then the game engine figures out which entities are impacted."

### What Was Done

#### New Modules Created
1. **engine/aoeGeometry.js** — Pure geometry module
   - `getEffectiveRadius(targeting)` — converts spell geometry to effective radius
   - `isInAoE(position, center, targeting)` — checks if a point is within an AoE
   - `computeOptimalCenter(caster, enemies, castRange, aoeRadius)` — finds best AoE placement
   - 27 tests covering cube, sphere, cone, cylinder, wall, edge cases

2. **engine/targetResolver.js** — Game engine target resolution
   - `resolveAoETargets(caster, spellDef, aoeCenter, allCombatants, options)` — authoritative "who gets hit" determination
   - Excludes caster, dead, same-side (unless friendly fire enabled)
   - 11 tests covering all AoE shapes, edge cases, friendly fire

#### Modified Files
3. **data/spells.js** — Structured targeting geometry
   - `'30ft cube'` → `{ shape: 'cube', size: 30 }`
   - `'20ft sphere'` → `{ shape: 'sphere', radius: 20 }`
   - `'60ft cone'` → `{ shape: 'cone', length: 60 }`
   - `'20ft radius, 40ft high cylinder'` → `{ shape: 'cylinder', radius: 20, height: 40 }`
   - Added `getAoERadius()` utility to registry API
   - Updated schema documentation

4. **engine/spellResolver.js** — Engine-resolved targeting
   - AoE branch now checks for `action.aoeCenter` and resolves targets via targetResolver
   - Falls back to `action.targets` for backward compatibility
   - Updated log formatting for structured shapes

5. **ai/tactics.js** — Intent-only actions
   - Replaced `getEnemiesInAoE()` with `planAoEPlacement()`
   - 5 AoE evaluators updated: `evalOpeningAoEDisable`, `evalRecastHypnoticPattern`, `evalMageFireball`, `evalArchmageConeOfCold`, `evalLichCloudkill`
   - All now return `aoeCenter: { x, y }` instead of `targets: [...]`

#### New Action Format
```javascript
// NEW — AI declares intent, engine resolves targets
{ spell: 'Hypnotic Pattern', level: 3, aoeCenter: { x: 10, y: 0 } }

// OLD — still supported for backward compat
{ spell: 'Hypnotic Pattern', level: 3, targets: [...] }
```

### Architecture Separation
| Layer | Responsibility |
|-------|---------------|
| AI (tactics.js) | Decides WHAT spell and WHERE to place it |
| Target Resolver | Decides WHO is affected (geometry check) |
| Spell Resolver | Resolves saves, damage, effects |

### Results
- **829 tests pass** (was 788, +41 new)
- 11 test suites, 0 failures
- Commit `ac54248`, pushed to main
- Clean backward compatibility maintained

### What Was Learned
- Separating "who decides" from "who computes" makes the system easier to test and reason about
- The structured geometry data (`{ shape: 'cube', size: 30 }`) is more useful than display strings for both the engine and AI layers
- Backward compatibility via fallback (`aoeCenter` preferred, `targets` still works) allows incremental migration
