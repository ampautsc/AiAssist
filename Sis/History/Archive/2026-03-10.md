## Interaction: Implement D&D 5e Combat AI Tactics Engine

### What Was Discussed
Creator requested implementation of a new `tactics.js` file for the D&D 5e combat simulation system. The file needed to pass all tests in `tactics.test.js` (151 tests covering battlefield assessment, evaluators, profiles, decision engine, reaction evaluators, and monster AI).

### What Was Done

#### tactics.js — Complete Rewrite
Implemented a priority-based tactical AI system with:

1. **`assessBattlefield(creature, allCombatants, round)`** — builds a context snapshot with enemies, allies, activeEnemies, enemiesInMelee, charmedAllies, hpPct, isFlying, isInvisible, canFly

2. **Targeting helpers** — `selectHighestThreat` (prefers casters), `selectWeakest` (lowest HP), `selectClosestCharmedAlly`

3. **Evaluator functions** for each archetype:
   - Lore Bard: survival invisibility, opening AoE, concentration management, self-heal bonus action
   - Cult Fanatic: invisible fallback, flying target, spiritual weapon, shake awake, melee/spell
   - Monster archetypes: dragon breath, giant rock throw/melee, mage fireball/firebolt/misty step, archmage cone of cold, lich power word stun/finger of death/cloudkill
   - Shared: dodge, ranged cantrip with approach

4. **Reaction evaluators**: cutting words (bardic inspiration), counterspell (dangerous spells), legendary resistance

5. **`makeDecision()`** — runs evaluators in priority order, collects `_bonusActionOnly` results and merges their bonus actions into the main decision

6. **Profile registry** — 10 profiles (lore_bard, cult_fanatic, generic_melee/ranged, dragon, giant_bruiser, mage_caster, archmage_caster, lich_caster, undead_melee)

7. **Factory functions** — `makeTacticalAI(profileMap)` and `makeReactionAI(profileMap)` for encounter runners

#### encounterRunner.js — Complete Rewrite
Discovered that the encounterRunner also needed a complete rewrite to support the new API format used by all tests:

- **New API**: `runEncounter({ combatants, getDecision, maxRounds, verbose })` → `{ winner, rounds, log, analytics, positionSnapshots }`
- New exports: `rollInitiative`, `resetTurnState`, `processStartOfTurn`, `processEndOfTurnSaves`, `checkVictory`, `resolveWeaponAttack`, `resolveBreathWeapon`, `resolveShakeAwake`, `resolveDragonFear`, `buildAnalytics`
- Proper concentration save checking on weapon hits (calls `mech.breakConcentration` on failed saves)
- Turn processing: initiative → reset state → start-of-turn effects → get decision → resolve action + bonus → end-of-turn saves → victory check
- Victory conditions: party/enemy dead, stalemate at round 15+ with incapacitated enemies, draw on timeout

### What Was Learned

**Key technical insights:**
- Vicious Mockery is in `cantrips` array, not `spellsKnown`, so `evalFallbackCantrip` must check both
- lich_caster profile needs `evalMageFireball` before `evalMageFireBolt` (not just FireBolt) — test expected Fireball fallback when high slots spent but level 3 still available
- The `_bonusActionOnly` pattern allows evaluators like `evalConcentrationSelfHeal` and `evalMageMistyStep` to augment another evaluator's main decision with a bonus action
- The encounterRunner was already being used with the new API in `scenarioHarness.js` but hadn't been rewritten yet

## Interaction: Architectural Refactor — Separate AI Intent from Engine Target Resolution

### What Was Discussed
Creator identified a fundamental architecture issue: the AI (tactics.js) was deciding WHO gets hit by AoE spells, when that should be the game engine's job. The AI should declare intent (WHAT spell, WHERE to center it), and the engine should determine WHO is affected based on geometry.

Creator's exact insight: "The spell really shouldn't get to decide who is impacted. The game asks what the entity wants to do on their turn, and the entity replies with Cast, Hypnotic Pattern, Centered on Hex68... Then the game engine figures out which entities are impacted."

### What Was Done

#### New Modules Created
1. **engine/aoeGeometry.js** — Pure geometry module
   - `getEffectiveRadius(targeting)` — converts spell geometry to effective radius
   - `isInAoE(position, center, targeting)` — checks if a point is within an AoE
   - `computeOptimalCenter(caster, enemies, castRange, aoeRadius)` — finds best AoE placement
   - 27 tests covering cube, sphere, cone, cylinder, wall, edge cases

2. **engine/targetResolver.js** — Game engine target resolution
   - `resolveAoETargets(caster, spellDef, aoeCenter, allCombatants, options)` — authoritative "who gets hit" determination
   - Excludes caster, dead, same-side (unless friendly fire enabled)
   - 11 tests covering all AoE shapes, edge cases, friendly fire

#### Modified Files
3. **data/spells.js** — Structured targeting geometry
   - `'30ft cube'` → `{ shape: 'cube', size: 30 }`
   - `'20ft sphere'` → `{ shape: 'sphere', radius: 20 }`
   - `'60ft cone'` → `{ shape: 'cone', length: 60 }`
   - `'20ft radius, 40ft high cylinder'` → `{ shape: 'cylinder', radius: 20, height: 40 }`
   - Added `getAoERadius()` utility to registry API
   - Updated schema documentation

4. **engine/spellResolver.js** — Engine-resolved targeting
   - AoE branch now checks for `action.aoeCenter` and resolves targets via targetResolver
   - Falls back to `action.targets` for backward compatibility
   - Updated log formatting for structured shapes

5. **ai/tactics.js** — Intent-only actions
   - Replaced `getEnemiesInAoE()` with `planAoEPlacement()`
   - 5 AoE evaluators updated: `evalOpeningAoEDisable`, `evalRecastHypnoticPattern`, `evalMageFireball`, `evalArchmageConeOfCold`, `evalLichCloudkill`
   - All now return `aoeCenter: { x, y }` instead of `targets: [...]`

#### New Action Format
```javascript
// NEW — AI declares intent, engine resolves targets
{ spell: 'Hypnotic Pattern', level: 3, aoeCenter: { x: 10, y: 0 } }

// OLD — still supported for backward compat
{ spell: 'Hypnotic Pattern', level: 3, targets: [...] }
```

### Architecture Separation
| Layer | Responsibility |
|-------|---------------|
| AI (tactics.js) | Decides WHAT spell and WHERE to place it |
| Target Resolver | Decides WHO is affected (geometry check) |
| Spell Resolver | Resolves saves, damage, effects |

### Results
- **829 tests pass** (was 788, +41 new)
- 11 test suites, 0 failures
- Commit `ac54248`, pushed to main
- Clean backward compatibility maintained

### What Was Learned
- Separating "who decides" from "who computes" makes the system easier to test and reason about
- The structured geometry data (`{ shape: 'cube', size: 30 }`) is more useful than display strings for both the engine and AI layers
- Backward compatibility via fallback (`aoeCenter` preferred, `targets` still works) allows incremental migration

## Interaction: Full Sweep — Fix Everything Found (3D Flight, Breath/Fear Engine, Data Consistency)

### What Was Discussed
Creator said "Start implementation" to execute the 15-step "Full Sweep" plan approved in the prior session. The plan covered 4 categories:
- **A**: 3D altitude in AoE calculations (flight-aware geometry)
- **B**: Breath weapon & dragon fear migration to engine-resolved targeting
- **C**: Data consistency fixes (gemFlight duration, flying tags, flySpeed/hasWingedBoots)
- **D**: Documentation/naming consistency

### What Was Done

#### Category A — 3D Altitude in AoE (Steps A1-A5)
1. **aoeGeometry.js** — Added `FLYING_ALTITUDE_FT = 30` constant. Updated `isInAoE()` to accept `options.flying`: sphere/cube/cone use 3D distance `sqrt(horizontal² + 30²)` vs radius; cylinder checks `height >= 30` AND horizontal distance. Added `canAoEReachFlying(targeting)` helper for AI to quickly check if an AoE can reach flying creatures.
2. **targetResolver.js** — Passes `{ flying: !!c.flying }` to `isInAoE()` for each combatant.
3. **tactics.js** — `planAoEPlacement()` now filters out flying enemies when `canAoEReachFlying()` returns false (small ground AoEs won't waste placement on unreachable targets).
4. **22 new aoeGeometry tests** — sphere/cube/cone/cylinder hit/miss flying creatures, canAoEReachFlying for all shapes.
5. **5 new targetResolver tests** — flying creatures included/excluded based on AoE geometry.

#### Category B — Breath Weapon & Dragon Fear Engine Resolution (Steps B6-B11)
6. **creatures.js + buildConverter.js** — Added `targeting` geometry to breath weapon (`{ type: 'area', shape: 'cone', length: 15/30 }`) and dragon fear (`{ type: 'area', shape: 'cone', length: 30 }`).
7. **encounterRunner.js** — `resolveBreathWeapon()` and `resolveDragonFear()` now support `aoeCenter` path: builds spellDef from creature's targeting data, calls `resolveAoETargets()` with friendly fire enabled, falls back to `action.targets` for backward compat.
8. **tactics.js** — `evalConcentrationBreathWeapon()`, `evalDragonBreathWeapon()`, and new `evalDragonFear()` all return `aoeCenter` instead of `targets`.
9. **evalDragonFear** — Brand new evaluator: checks dragonFear resource available, picks optimal placement via `computeOptimalCenter`, returns `{ action: 'dragonFear', aoeCenter }`. Wired into lore_bard (after opening AoE) and dragon (after breath weapon) profiles.
10. **4 new encounterRunner tests** — engine-resolved breath weapon and dragon fear with aoeCenter.
11. **4 new tactics tests** — dragon fear evaluator with/without resource, placement, profile wiring.

#### Category C — Data Consistency Fixes (Steps C12-C14)
12. **buildConverter.js** — Fixed `gemFlight.duration: 10` → `gemFlight.maxRounds: 10` (matches what mechanics.js expects).
13. **creatures.js** — Factory now checks `if (t.tags?.includes('flying')) creature.flying = true`. Young Red Dragon now starts airborne.
14. **buildConverter.js** — Species fly speed (`speciesFlySpeed`) or Winged Boots (`hasWingedBoots`) now set `creature.flying = true` at end of conversion.

#### Category D — Naming Consistency
15. All targeting geometry uses consistent `{ type: 'area', shape: '...', ... }` format across spells, breath weapons, and dragon fear.

### Results
- **869 tests pass** (was 829, +40 new across 5 test files)
- 11 test suites, 0 failures
- Commit `88c2fb4`, pushed to main
- All 15 steps of the Full Sweep plan complete

### What Was Learned
- 3D distance calculation for flight is straightforward: `sqrt(horizontal² + altitude²)` for most shapes, but cylinders need a separate height check since they have explicit height bounds
- `canAoEReachFlying()` is a valuable optimization — the AI can skip flying enemies entirely for small ground-level AoEs rather than wasting computation on optimal placement
- The `aoeCenter` pattern from the prior refactor (spells) extended naturally to breath weapons and dragon fear — same pattern, same fallback mechanism
- Data consistency between buildConverter and creatures factories was silently broken (gemFlight.duration vs maxRounds) — the test-first approach caught this

### Decisions Made
- Young Red Dragon starts flying (has `tags: ['flying']`) — this is the intended behavior for a dragon
- Gem flight (Amethyst Dragon bard) still requires activation (separate action, limited uses) — different from natural flight
- `flySpeed` and `hasWingedBoots` both grant immediate `flying: true` — in combat, characters with flight capability would typically be airborne

## Interaction: Combat Log Audit — Found and Fixed 2 More Bugs

### What Was Discussed
Creator asked to "double check all combat logs looking for abnormalities" after the previous 8 bug fixes. Conducted systematic audit of 303,200 encounters across 379 builds × 8 scenarios.

### Anomalies Investigated

1. **`+undefined` in save log display** (Bug A) — Every saving throw log showed `d20:5+undefined=5` instead of `d20:5+0=5`. Root cause: `makeSavingThrow()` in `mechanics.js` computed `total = result + mod` correctly but didn't include `saveBonus: mod` in its return object. Log lines referenced `save.saveBonus` which was `undefined`. **Calculation was correct, only display was broken.**

2. **Bard Dodges while all enemies disabled** (Bug B) — `evalConcentrationAllDisabled` in `tactics.js` made the bard take Dodge action when all enemies were incapacitated by Hypnotic Pattern. This wasted 10 rounds doing nothing until the spell expired. Affected 16% of all runs (1,753 out of 10,896 audited). The bard should instead use Vicious Mockery to chip down enemies.

3. **Flying bard vs melee-only enemies = stalemate** — NOT a bug. Melee enemies (werewolves) correctly can't reach a bard at 30ft altitude. The bard's VM can't kill 4×58 HP = 232 HP of werewolves in 20 rounds. This is physically correct and the scoring appropriately rewards the stalemate at 25 points (vs wins at 100).

4. **0-damage bard (1,468 rankings)** — NOT a bug. CC spells (Hypnotic Pattern, Hold Person) don't deal direct damage. Bard uses VM when available but CC-focused casting is correct strategy.

5. **0-spells bard (21 cases)** — NOT a bug. Bard killed in round 1 before acting (lost initiative vs Frost Giant/Dragon). Expected outcome.

### Fixes Applied

**Bug A Fix: `makeSavingThrow` returns `saveBonus`** — [mechanics.js](DnD/dnd-builder/server/combat/engine/mechanics.js#L294)
- Added `saveBonus: mod` to the return object
- Now consistent with `concentrationSave` which already returned `saveBonus`
- All log strings now correctly show `+0`, `+3`, `+-1` etc.

**Bug B Fix: `evalConcentrationAllDisabled` attacks instead of dodging** — [tactics.js](DnD/dnd-builder/server/combat/ai/tactics.js#L147)
- Changed from `{ type: 'dodge' }` to `{ spell: 'Vicious Mockery', target: selectWeakest(helplessEnemies) }`
- Targets the weakest incapacitated enemy with VM
- For HP-charmed targets, damage breaks charm on THAT creature only — others stay incapacitated
- For flying bards vs melee: woken melee enemy still can't reach, so near-zero risk

### Test Changes
- Added `saveBonus` assertion to `makeSavingThrow` tests in mechanics.test.js (+1 new test)
- Updated `evalConcentrationAllDisabled` tests in tactics.test.js — now expects VM on weakest instead of Dodge (+2 new tests, 1 updated)
- Updated `makeDecision` integration test to expect VM instead of Dodge
- **795 tests pass, 0 failures**

### Simulation Results (Record ID: 69b0e2f6aea5444bf7983fb5)
- 379 builds × 8 scenarios × 100 sims = 303,200 encounters in 120s
- **+undefined in saves: 0** (was in every save log before)
- **Dodge-heavy runs: 310** (down from 1,753 — 82% reduction, remaining 19 are legitimate edge cases in Cult Fanatics)
- **Stalemate rate: 4.3%** (down from 6.3%)

New Top 5:
1. Aasimar — Armored Escapist — 9.07 avg

3. Metallic Dragonborn — Dragon Fear Caster — 8.62 avg

### What Was Learned
- The `evalConcentrationAllDisabled` dodge bug was responsible for 16% of wasted combat turns — the bard would dodge for up to 10 rounds while all enemies were frozen, accomplishing nothing
- Display bugs (`+undefined`) can mask real values — the saves were CALCULATED correctly but logs looked wrong, which made debugging harder
- Flying bard vs melee stalemates are a legitimate D&D outcome, not a bug — the scoring system appropriately rewards partial survival
- 0-damage bards are expected for CC-focused builds — Hypnotic Pattern's value is disabling enemies, not dealing damage
