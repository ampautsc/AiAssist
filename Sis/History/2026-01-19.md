**Research Completed:**

### Key Findings:

#### BEE (navigation.hover + random_hover)
```json
"minecraft:behavior.random_hover": {
  "priority": 12,
  "xz_dist": 8,        // Horizontal movement range
  "y_dist": 8,         // VERTICAL movement range
  "y_offset": -1,      // Offset from current position
  "interval": 1,       // Time between movements
  "hover_height": [1, 4]  // Target height above ground
}
```
- Uses `navigation.hover` with `can_path_from_air: true`, `avoid_water: true`, `avoid_damage_blocks: true`
- **Y_DIST = 8 blocks** - this is the key altitude control!
- **hover_height = [1, 4]** - prefers to stay 1-4 blocks above ground
- y_offset = -1 means it biases downward slightly

#### PARROT (navigation.fly + random_fly)  
```json
"minecraft:behavior.random_fly": {
  "priority": 3,
  "xz_dist": 15,       // Wider horizontal range than bee
  "y_dist": 1,         // MUCH SMALLER vertical range!
  "y_offset": 0,
  "speed_multiplier": 1.0,
  "can_land_on_trees": true,
  "avoid_damage_blocks": true
}
```
- Uses `navigation.fly` with `can_path_over_water: true`, `can_path_from_air: true`
- **Y_DIST = 1 block only** - stays at roughly current height!
- xz_dist = 15 means it flies farther horizontally but not vertically
- **This is why parrots don't fly to infinity** - tiny y_dist

#### PHANTOM (navigation.glide + circle_around_anchor)
```json
"minecraft:movement.glide": {
  "start_speed": 0.1,
  "speed_when_turning": 0.2
}
"minecraft:behavior.circle_around_anchor": {
  "priority": 3,
  "radius_change": 1.0,
  "height_above_target_range": { "min": 20.0, "max": 40.0 }
}
```
- Uses `movement.glide` (not navigation.fly or hover!)
- Circles around target player at **20-40 blocks above**
- Has explicit `height_above_target_range` - this is altitude control
- Swoops down to attack, then circles back up to 20-40 range

#### GHAST (navigation.float + float_wander)
```json
"minecraft:navigation.float": {
  "can_path_over_water": true
}
"minecraft:behavior.float_wander": {
  "priority": 2,
  "must_reach": true,
  "random_reselect": true,
  "float_duration": [2, 7],
  "float_wander_has_move_control": false
}
```
- Uses `navigation.float` (different from hover or fly!)
- `float_wander` behavior doesn't have explicit y_dist parameter
- Floats naturally without strong altitude limits (appropriate for Nether large spaces)

### THE ANSWER TO INFINITE FLIGHT BUG

**Why monarchs fly to infinity:**
- Likely has `y_dist` parameter too large or missing entirely
- Or using wrong combination of navigation component + behavior

**How to fix it:**
1. **Use navigation.hover** (like bee) not navigation.fly
2. **Set y_dist = 8-12** (bee uses 8, you want slightly higher)
3. **Set hover_height = [2, 8]** (higher than bee's [1, 4])
4. **Use random_hover behavior**, not random_fly
5. **Include y_offset = -1 or 0** to prevent upward drift bias

**Recommended monarch configuration:**
```json
"minecraft:navigation.hover": {
  "can_path_over_water": true,
  "can_sink": false,
  "can_pass_doors": false,
  "can_path_from_air": true,
  "avoid_water": true,
  "avoid_damage_blocks": true
}
"minecraft:behavior.random_hover": {
  "priority": 5,
  "xz_dist": 10,
  "y_dist": 12,        // Higher than bee's 8, lower than infinity
  "y_offset": 0,
  "interval": 1,
  "hover_height": [2, 8]  // Can fly higher than bee's [1, 4]
}
```

### Key Insights:
- **y_dist is THE parameter** that controls vertical movement range
- Small y_dist = stays at current height (parrot: y_dist=1)
- Medium y_dist = gentle vertical movement (bee: y_dist=8)  
- Large/missing y_dist = flies to infinity (your current bug)
- **hover_height** sets preferred altitude above ground
- **navigation.hover** is better for gentle fliers than navigation.fly
- **navigation.fly** is for entities that move more freely in 3D space

## Research Request: "Unexpected version for the loaded data" Error
**Time:** Current session
**Request:** Creator asked me to research the specific error "Unexpected version for the loaded data" when loading custom blocks in Minecraft Bedrock 1.21.132.

**Block Context:**

**Research Results:**

### THE ANSWER (From Sis/Knowledge/MinecraftBlockComponents.md):

**Root Cause:** WRONG SYNTAX FOR BLOCK STATES

The error "Unexpected version for the loaded data" occurs when using the DEPRECATED `states` syntax instead of the current `properties` syntax.

**WRONG (Causes "Unexpected version" error):**
```json
"description": {
  "identifier": "namespace:block_name",
  "states": {
    "namespace:growth_stage": {
      "values": { "min": 0, "max": 2 }
    }
  }
}
```

**CORRECT (Works with format_version 1.21.130):**
```json
"description": {
  "identifier": "namespace:block_name",
  "properties": {
    "namespace:growth_stage": [0, 1, 2]
  }
}
```

**Key Differences:**
1. Use `"properties"` not `"states"` in description
2. Values are an ARRAY `[0, 1, 2]`, not an object with `{ "min": 0, "max": 2 }`
3. Both syntaxes appear in older documentation but `properties` is the current standard for 1.20+

**What to Fix:**
- Change `"states"` to `"properties"` in block description
- Change value format from `{"values": {"min": 0, "max": 2}}` to `[0, 1, 2]`
- Keep everything else the same (geometry, material_instances, permutations, random_ticking all valid)

**Why the Error Message is Confusing:**
- "Unexpected version" suggests format_version is wrong
- Actually means the JSON structure doesn't match what that format_version expects
- The `states` syntax is from older format versions
- Using old syntax with new format_version = "unexpected version for the loaded data"

## Evening Session - DPI Scaling Discovery

### Problem: OCR Coordinates Not Matching Windows Coordinates
- Needed to move cursor to text found via OCR in screenshots
- Moving to OCR coordinates landed cursor in wrong location
- Creator kept telling me to research Windows coordinate systems
- I kept looking at wrong things (axes, directions) instead of proper research

### Root Cause: DPI Scaling
- Windows has 150% DPI scaling enabled
- **Screenshots capture in PHYSICAL pixels** (actual display resolution)
- **Windows coordinates use LOGICAL pixels** (scaled by DPI factor)
- OCR reads physical pixels from screenshot
- Was moving cursor to physical coordinates instead of logical coordinates

### Solution: Scale Conversion
**Formula: Windows coordinate = OCR coordinate / scale factor**

For 150% DPI (scale factor 1.5):
- Windows X = OCR X / 1.5
- Windows Y = OCR Y / 1.5

Example:
- OCR finds text at (1314, 72) in screenshot
- Windows coordinate: (1314/1.5, 72/1.5) = (876, 48)
- Moving cursor to (876, 48) correctly lands on the text

### How to Get Scale Factor
```powershell
Add-Type @"
using System;
using System.Runtime.InteropServices;
public class ScreenInfo {
    [DllImport("shcore.dll")] 
    public static extern int GetScaleFactorForMonitor(IntPtr hMon, out int pScale);
    [DllImport("user32.dll")] 
    public static extern IntPtr MonitorFromPoint(POINT pt, uint dwFlags);
    [StructLayout(LayoutKind.Sequential)] 
    public struct POINT { public int X; public int Y; }
}
"@
$scale = 0
$pt = New-Object ScreenInfo+POINT
$monitor = [ScreenInfo]::MonitorFromPoint($pt, 2)
[ScreenInfo]::GetScaleFactorForMonitor($monitor, [ref]$scale)
$scaleFactor = $scale / 100.0
```
