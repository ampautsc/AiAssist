## DnD Multiattack Fixes (T-Rex & Giant Ape)

### What Happened
T-Rex multiattack was broken — it used a single "Multiattack" action instead of distinct Bite+Tail attacks on different targets. Giant Ape was also broken — it allowed both Fist and Rock as multiattack options instead of restricting multiattack to Fist×2.

### Root Cause
No `multiattackWeapons` field existed to distinguish between:
- **Distinct-weapon multiattack** (T-Rex: Bite + Tail on different targets)
- **Same-weapon multiattack** (Giant Ape: Fist × 2 on same/different targets, Rock is separate)

### Fix
- Added `multiattackWeapons` to spell data: T-Rex `['Bite','Tail']`, Giant Ape `['Fist','Fist']`
- Updated TurnMenu to use `multiattackWeapons` for weapon selection in multiattack menus
- Updated ActionResolver to propagate `multiattackWeapons` through polymorph application, prePolymorphState saves, and revert paths
- All 380 tests passing

### What Was Learned
The polymorph application path didn't copy all beast form fields to the combatant — `multiattackWeapons` was silently dropped. Any new beast form field must be explicitly propagated through all three polymorph code paths: self-target update, prePolymorphState save (both enemy and self), and revert.

## Image Generation Research — Character Consistency for DnD Encounter Screen

### The Problem
Boss wants engaging scene-specific imagery for the encounter screen. Characters (PCs and NPCs) need to look consistent across different generated scenes. Sometimes multiple characters in one scene.

### Phase 1 — Initial Research (Broad Comparison)
Evaluated 4 platforms. Initially recommended OpenAI GPT Image 1.5 based on feature documentation.

### Phase 2 — Boss Pushback & Deep Dive
Boss correctly challenged the recommendation: "Open AI has excellent image generation, but in my experience it has not done great at incorporating existing images." Demanded evidence, not feature lists.

### Revised Evidence-Based Findings

**OpenAI GPT Image 1.5 — UNPROVEN for character consistency**
- `input_fidelity: "high"` is documented but new and untested by us
- OpenAI's OWN docs admit: "may occasionally struggle to maintain visual consistency for recurring characters"
- Boss's personal experience contradicts the marketing claims
- No concrete evidence gathered proving it works for identity preservation

**Stability AI — NOT A CONTENDER**
- No native character identity preservation feature. Generic img2img only.

**Replicate Ecosystem — STRONG EVIDENCE (purpose-built models)**

1. **Google Nano-Banana** (91.7M runs, Official Google model)
   - Explicitly designed for "Character and Style Consistency" per README
   - Multi-image fusion: blend up to 3 reference images into new scenes
   - Google admits limitation: "may not be 100% reliable in all scenarios"
   - $0.039/image, ~10 seconds

2. **InstantID** (6.7M runs, 1+ year old)
   - Purpose-built face identity preservation via dual IdentityNet + IP-Adapter pipeline
   - Explicit `identitynet_strength_ratio` and `ip_adapter_scale` controls
   - Multiple SDXL backbones (photorealism, anime, fantasy)
   - $0.0015/image — cheapest by far

3. **Flux Kontext Pro** (34M runs, Official Black Forest Labs)
   - Subject-preserving image editing via text instructions
   - Best for modifying existing character images while keeping identity
   - ~$0.03/image

4. **Flux-PuLID** (2.5M runs, ByteDance)
   - Face reference + prompt → identity-consistent images
   - `id_weight` control (0-3) for identity strength
   - Note: "might not work as well for some male faces"

**Anthropic/Claude** — No image generation. Vision only.

### Revised Recommendation
**Replicate ecosystem is the clear winner for character consistency.** The models are purpose-built for exactly this task, unlike OpenAI's general-purpose approach.

Proposed workflow:
1. Character creation portraits: InstantID ($0.0015/ea)
2. Scene generation with characters: Nano-Banana ($0.039/ea)
3. Multi-character scenes: Nano-Banana multi-image fusion
4. Refinement: Nano-Banana conversational editing

Estimated cost per encounter: ~$0.08-0.15

### Key Lesson
**Don't recommend based on feature documentation. Recommend based on evidence and purpose-built design.** OpenAI has the best marketing; Replicate has the best tools for this specific job. Boss's instinct was right.

## Learning Protocol Rewrite — Making It Enforceable

### What Was Wrong
The first version of the learning protocol was aspirational prose. It used soft language: "pause and run this loop," "keep mental notes," "should this become a skill?" Meanwhile Rules #1 and #2 say "ABSOLUTELY FORBIDDEN" and "REQUIRED BEHAVIOR." Learning was a suggestion sitting next to mandates. Of course it got skipped.

The skill creation threshold was set at "3 or more occurrences" — giving a permanent excuse to never create a skill the first time something is done well. Boss was right: if you solved a problem with clear steps, capture it NOW.

### What Changed
1. **Added Rule #3 to sis-core** — "LEARN FROM EVERYTHING" with the same ABSOLUTELY FORBIDDEN / REQUIRED BEHAVIOR structure as Rules #1 and #2
2. **Added Rule #3 to the mandatory pre-action check** — now every action requires checking the index first
3. **Added skill creation + index verification to mandatory close-out** — not optional, not "if you feel like it"
4. **Dropped the "3 occurrences" threshold** — if it has clear steps, it's a skill NOW
5. **Rewrote learning-protocol.instructions.md** — stripped the soft language, made it direct references to Rule #3, MANDATORY labels on every section
6. **Removed the flowchart** — pretty diagrams don't get followed. Short imperative instructions do.

### The Meta-Lesson
Instructions that don't match the enforcement level of the rules they sit beside will be ignored. If you want something followed, give it the same teeth as everything else.

## Dice Architecture Redesign — Boss Requirements (CRITICAL)

### Background
Boss asked me to explain how the seeded dice system works. Through questioning, he exposed that the current implementation is fundamentally broken — the seeded commit-reveal protocol only covers the player's own action rolls. Enemy attack rolls, saving throws triggered by enemy actions, concentration saves, etc. all bypass the seed and use plain Math.random(). This violates the core requirement: **the user's physical interaction with the dice contributes luck to every roll.**

### The Actual Requirement (Boss's Words)
Every dice roll in the game — player's, enemy's, saving throws, damage, everything — must be seeded by the user's click interaction. The engine never rolls dice on its own. Period.

### Correct Architecture

**Engine side:**
- Engine reaches a point needing dice (any roll: attack, damage, save, concentration, death save, initiative...)
- Emits a **dice request**: `{ reason: "Goblin Claw damage", dice: [d6, d6, d6], modifier: +3, owner: "goblin" }`
- Pauses. Waits for results.
- Results come back: `[4, 2, 6]` → engine computes total (4+2+6+3=15), applies damage, continues.

**UI side — DiceArena:**
- Opens showing all requested dice lined up on a "tray" (e.g., three d6 cubes visible)
- Context shown: "Goblin Claw damage — 3d6+3"
- **Player's rolls**: User clicks each die one at a time. Click → seed (ms timestamp) → sent to engine → engine returns result → die animates rolling → lands on number → result appears where die was sitting on tray
- **Enemy rolls**: Same visual display, but dice auto-roll one by one (no click). User watches the goblin's dice tumble. Display only.
- After all dice resolved: total displayed, arena closes (or transitions to next roll), engine continues

**Key design rules:**
1. One engine pause per dice-needing event
2. All dice for that event visible at once in the arena
3. Player clicks each die individually (one seed per die per click)
4. Enemy dice auto-animate (server provides values, no user click)
5. User sees ALL rolls — theirs AND the enemy's — same visual treatment
6. Engine is a coroutine/state machine that yields on dice needs

### What This Kills in Current Code
- `executeAiTurns()` auto-running enemy turns silently — enemy turns must pause for dice display
- The commit-reveal HMAC-SHA256 protocol in dice.js — overkill for the actual requirement (timestamp seed per click)
- The batch `confirmRolls(clientSeed)` API — wrong granularity (one seed for all rolls vs one seed per die)
- `_seededState` module-level singleton — wrong model entirely

### What Surprised Me
I confidently described the existing architecture as "correctly wired" without understanding Boss's actual requirement. The system was internally consistent but built to the wrong spec. I should have asked "what does 'seeded by user interaction' mean for enemy turns?" on day one.

### Status
Design requirements captured. Implementation not started. This is a significant refactor of the engine-to-UI contract.
