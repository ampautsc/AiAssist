## Interaction 3 - Click Breakthrough

### What Was Discussed
Creator kept telling me: "you didn't click", "no click", "activate minecraft", "REMEMBER". I was moving the cursor but not actually clicking. Tried multiple Win32 API approaches - none worked until I found the right combination.

### What Was Learned
**The Click Problem:**
- mouse_event(0x0002, 0, 0, 0, 0) for LEFTDOWN was being called
- mouse_event(0x0004, 0, 0, 0, 0) for LEFTUP was being called
- Cursor was moving to the right position
- But no actual click was registering in Minecraft

**Why It Failed:**
- Basic mouse_event calls don't work reliably with UWP apps like Minecraft
- Need proper focus management + specific calling convention
- VS Code kept stealing focus back

**What Finally Worked (WindowsFormsClick.ps1):**
```csharp
[DllImport("user32.dll", CharSet = CharSet.Auto, CallingConvention = CallingConvention.StdCall)]
public static extern void mouse_event(long dwFlags, long dx, long dy, long cButtons, long dwExtraInfo);
```
Key differences:
- CharSet = CharSet.Auto
- CallingConvention = CallingConvention.StdCall  
- long parameters instead of uint/int
- Proper focus with SetForegroundWindow BEFORE moving cursor
- Adequate delays (500ms after focus, 200ms after cursor move, 100ms between down/up)
- Running in separate PowerShell process so VS Code doesn't steal focus

**Success:** Clicked Play button → Reached worlds list! OCR confirmed: world, realm, creative, survival, play all detected.

### Decisions Made

- Run click scripts in separate PowerShell process to avoid focus stealing
- Always: SetForegroundWindow → wait → SetCursorPos → wait → click → wait
- Delays matter: 500ms, 200ms, 100ms minimum at each step
